# Work Summary — 月度工作总结生成器

根据多个项目的 git commit 记录，自动归纳整理出结构化的工作总结，按项目分组、按功能模块归类，输出为 Markdown 文件。

## 工作流程

### Step 1：确定参数

从用户的输入中解析以下参数，缺失的用默认值或询问用户：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| **项目路径** | 要汇总的项目目录列表 | 无默认值——必须由用户提供或确认 |
| **时间范围** | 月份（如 `2026-04`）或自定义区间 | 当前月份 |
| **git 作者** | 只统计当前用户的提交 | 自动通过 `git config user.name` 获取，不询问 |
| **输出路径** | .md 文件保存位置 | 用户所选的第一个项目的根目录 |

**关于项目路径**：用户的项目分散在不同位置，不一定在同一个父目录下。你需要：
1. 如果用户直接给了路径列表，用它们
2. 如果用户说"扫描某个目录"，用 `find <dir> -maxdepth 2 -name ".git" -type d` 找出所有 git 仓库
3. 如果用户没给路径，**必须询问**——不要猜测，不要扫描用户未授权的目录

**关于时间范围**：
- "本月" / "这个月" → 当前月份第一天到今天
- "上个月" / "上月" → 上个月的 1 号到最后一天
- "2026-04" → 2026 年 4 月整月
- "最近两周" → 过去 14 天
- 模糊表达需要转换为具体的 `--since` 和 `--until` 参数

### Step 2：提取 Git 信息（两层数据）

commit message 经常写得不准确——可能含糊（"update"、"fix"）、可能跟实际改动对不上、可能一个 commit 塞了多件事。所以不能只看 message，需要结合实际文件变更来判断。

**第一层：提取 commit 列表（必做）**

```bash
git -C <project-path> log --since="YYYY-MM-01" --until="YYYY-MM+1-01" --author="<author>" --pretty=format:"%h|%ad|%s" --date=short --no-merges
```

**第二层：提取文件变更概览（必做）**

```bash
git -C <project-path> log --since="YYYY-MM-01" --until="YYYY-MM+1-01" --author="<author>" --pretty=format:"===COMMIT=== %h|%s" --stat --no-merges
```

`--stat` 会列出每个 commit 实际改了哪些文件、增删了多少行。这是判断"这个 commit 到底做了什么"的关键依据。

**关键点**：
- 使用 `--no-merges` 排除合并提交
- 如果某个项目在该月没有提交，跳过它
- 提取时记录项目名称（目录名或 `git remote get-url origin` 的仓库名）
- 如果某个路径不是 git 仓库或不存在，发出警告但继续处理其他项目

**第三层：针对含糊 commit 深入查看（按需）**

当遇到以下情况时，对该 commit 额外执行 `git diff` 查看实际改动内容：
- commit message 是含糊词："update"、"fix"、"修改"、"调整"、"wip"、"temp"、"."、"1" 等
- commit message 跟 `--stat` 显示的文件路径明显对不上（比如 message 说"修复登录"但改的文件全在订单模块下）
- 一个 commit 改了大量不相关文件（可能是把多件事塞在一个 commit 里了）

```bash
git -C <project-path> show <commit-hash> --stat
git -C <project-path> diff <commit-hash>~1 <commit-hash> -- <specific-file>
```

只在需要时才执行第三层，不要对每个 commit 都跑 diff——那样太慢了。

### Step 3：归纳整理（以实际变更为准）

这是最核心的步骤——综合 commit message 和实际文件变更，提炼出准确的工作总结。

**判断真实工作内容的优先级**：
1. **commit message 清晰且与 --stat 文件路径吻合** → 直接采信 message
2. **commit message 含糊但 --stat 文件路径能看出模块** → 根据文件路径所在的目录/模块来判断做了什么（比如改了 `src/views/login/` 下的文件就是登录模块相关）
3. **message 和 --stat 都看不出来** → 执行第三层 diff 查看具体改动，根据代码变化推断
4. **还是判断不了** → 保留该条目，标注为实际路径信息，不要丢弃

**核心思路——"模块在前，做了什么在后"**：

每条输出的结构是 `模块名-功能点  在这个模块里具体做了什么`。所以归纳的第一步是从文件变更中识别出"改的是哪个模块"，第二步才是总结"在这个模块里做了什么事"。

识别模块的方法：
- **看文件路径的目录结构**：`src/views/geo/console/` → GEO 控制台模块；`src/views/store/` → 门店模块
- **看组件/页面命名**：`TaskList.vue`、`AccountManage.vue` 等文件名本身就暗示了模块归属
- **看路由配置**：如果项目有 `router/` 目录，路由路径能反映模块划分
- **commit message 做辅助**：message 里提到的模块名可以辅助确认，但如果跟文件路径矛盾，以文件路径为准

**归纳原则**：
1. **先定模块再写内容**：先根据 --stat 文件路径确定这些 commit 属于哪个模块，再归纳这个模块里做了什么
2. **宁可拆细不要合粗——这是最重要的原则**：一个模块如果包含多个独立的子功能/子页面，每个子功能必须单独成行，不要折叠到一条里。判断"是否是独立子功能"的标准：
   - 对应不同的页面/路由/Tab → 必须拆
   - 对应不同的文件目录（如 `task-create/`、`task-detail/`、`task-list/` 是三个独立功能） → 必须拆
   - 功能之间有独立的操作流程（如"创建任务"和"管理任务"是两件事） → 必须拆
   - 只有真正重复的 commit（如同一个按钮先加后改样式再修 bug）才合并为一条

   **错误示范**（过度合并）：
   ```
   GEO-任务  任务创建、拓展问题生成、文章配置、发布配置、任务管理、任务详情
   ```

   **正确示范**（每个子功能独立一行）：
   ```
   GEO-任务管理  任务列表展示、状态识别、条件筛选、查看详情与删除
   GEO-任务创建  关键词配置、目标转化词配置与表单校验
   GEO-任务创建-拓展问题生成  AI一键拓词、手动编辑与拓词工具组合生成
   GEO-任务创建-文章生成配置  生成数量设置与配图上传管理
   GEO-任务创建-发布配置  发布平台选择、账号选择、去重策略与发布频率设置
   GEO-任务详情-文章管理与发布  文章预览、编辑、删除、立即发布与失败重发
   GEO-任务详情  任务信息汇总、拓展问题查看与今日发布计划展示
   ```

3. **用业务语言描述**：输出中不出现文件名、函数名、CSS 属性等技术细节。"整合功能入口、流程指引、数据趋势与账号状态展示" 而非 "added Dashboard.vue component with echarts bindings"
4. **保留关键信息**：修复了什么问题、完成了什么功能、给哪个客户做了什么——这些要保留
5. **合并重复**：仅限同一个功能点的重复 commit（如 fix typo、code review 修改、样式微调）合为一条。不同功能点不要合并
6. **客户专项单独列出**：如果 commit 涉及特定客户/网站的定制需求或问题处理，以「客户名  问题描述」的格式单独成行
7. **拆分大杂烩 commit**：如果一个 commit 的 --stat 显示改了多个不相关模块的文件，应拆开归入各自模块，而非合为一条
8. **模块命名要用业务名称**：用"GEO-任务管理"、"全域推-门店管理"这样的业务名称，不要用 "task-list module"、"store component" 这样的技术名称。如果从文件路径无法直接推断业务名称，看页面标题、路由 meta、面包屑等 UI 文案来确定

### Step 4：生成输出文件

输出为 Markdown 文件。**必须严格遵循以下格式**——这是用户给领导汇报的实际格式。

**Markdown 换行关键规则**：
生成的 .md 文件必须保证用 Markdown 编辑器或记事本打开后**每条工作内容独占一行、视觉上有清晰的换行**。具体做法：
- 每个项目标题（如 `1、AIGC`）前面空一行
- 每条工作条目之间**必须有一个空行**（即两条内容之间隔一个空行），这样 Markdown 渲染器才会断行
- 或者也可以在每行末尾加两个空格（Markdown 的硬换行语法），但空行方案更可靠
- **绝对不能**把所有条目连在一起不加空行——那样 Markdown 会渲染成一整段

**格式规则**：
- 顶层按项目编号分组：`1、项目名`、`2、项目名`、`3、项目名`
- 每个项目下的工作条目直接列出，不需要子编号
- 条目格式为：`模块名-功能名  功能描述`（模块名和功能名用短横线连接，功能名和描述之间用两个空格分隔）
- 功能描述中的多个要点用中文顿号「、」连接
- 客户专项条目格式为：`客户全称  问题描述+处理动作`
- 通用功能修复条目格式为：`客户端/插件名：具体描述`
- 不加 [新功能]、[Bug修复] 等类型标签
- 不写项目小结、不写整体总结、不写统计头信息
- 保持简洁紧凑，一条占一行

**输出模板（严格参照，注意每条之间有空行）**：

```
1、项目名称A

模块A-功能点1  功能描述，用顿号分隔多个要点

模块A-功能点2  功能描述

模块B-功能点1  功能描述

客户端：某个通用功能的修复或调整描述

XX公司/客户名  具体问题处理描述

2、项目名称B

客户端：功能描述

插件【插件名】布局xx  问题描述及处理

XX客户名  问题排查及处理描述

3、项目名称C

功能描述

客户名  问题处理描述
```

**真实示例（用户实际的汇报格式，每条之间有空行保证换行）**：


### Step 5：保存文件并告知用户

- 默认文件名格式：`工作总结_YYYY年MM月.md`
- **默认保存到用户所选的第一个项目的根目录下**；如果指定了多个项目，保存到第一个项目的根目录；用户明确指定了其他路径则按用户指定的来
- 保存后告知用户文件路径和涉及了多少个项目

## 注意事项

- **隐私**：commit message 可能包含敏感信息（如内部项目代号、客户名称），总结中的表述需要适当概括，不要原样暴露所有 commit message
- **编码**：Windows 下 git log 可能输出 GBK 编码的中文，必要时处理编码问题
- **大量提交**：如果某个项目单月提交量很大（>100），归纳时更要做好聚合，不要逐条列出
- **作者识别**：始终通过 `git config user.name` 自动获取当前用户名作为 `--author` 参数，只统计本人提交，不询问用户
